Our Process

How an engagement actually
runs at this firm.

We publish our methodology rather than abbreviate it. The six steps below describe what happens between the first call and the day we hand the system over. If we propose to do anything materially different, we'll say so in writing before kickoff.

01

Discovery & Requirements

1 – 2 weeks · written brief

Every engagement begins with discovery. We interview the people who will use, build, or pay for the system, map the current state, and identify the constraints that are easy to miss in a sales conversation — regulatory, organizational, integration, and timing.

  • Stakeholder interviews (typically 4 – 8 people)
  • System and data-flow mapping of the current state
  • Constraint inventory: regulatory, integration, organizational
  • Risk register, ranked by likelihood and impact
  • Written brief (15 – 40 pages) signed off before any code
02

Architecture & Planning

1 – 3 weeks · written plan

With the brief approved, we produce an architecture document and a delivery plan. Architecture decisions are made explicitly and trade-offs are written down. The delivery plan sequences work into vertical slices we can demo end-to-end, so progress is observable from week three onward.

  • System architecture document with trade-offs named
  • Data model, API surface, and integration contracts
  • Security model: authn, authz, secrets, data classification
  • Delivery plan, milestone schedule, and review cadence
  • Cost-and-effort estimate with confidence intervals
03

Iterative Development

Ongoing · weekly demos

We build in one-week iterations. Each Friday includes a live demo of working software in a staging environment, a written status report covering progress, risks, and decisions needed, and an updated backlog you can read in five minutes. The shared backlog is editable by the client; nothing about the project is hidden from you.

  • Weekly Friday demos in a real staging environment
  • Written status reports — progress, risks, decisions needed
  • Shared, editable backlog with effort estimates
  • Pull-request review process with documented standards
  • Continuous integration: tests, lint, type checks, security scans
04

QA, Security Review & Documentation

2 – 4 weeks · pre-launch

Before any system goes to production we run a structured QA pass, a security review, and a documentation sweep. None of these are afterthoughts — they have dedicated time in the schedule and dedicated deliverables you can audit.

  • Functional and regression QA against the original brief
  • Performance and load testing where appropriate
  • Internal security review: authn, authz, data exposure, dependencies
  • Operational documentation: runbooks, on-call notes, recovery procedures
  • Developer documentation: architecture, conventions, onboarding
05

Deployment & Handoff

1 – 2 weeks · cutover

Production deployment is a sequenced event, not a button-press. We deploy to a controlled environment, run validation against real data where possible, and cut over with a rollback path that's been tested. Handoff includes credentials, source control, infrastructure access, and a working session with whoever will own the system on your side.

  • Production deployment with documented rollback plan
  • Validation against production data or staged equivalents
  • Transfer of source code, secrets, and infrastructure access
  • Live handoff session with your engineering / ops team
  • Post-deployment monitoring window (typically 2 – 4 weeks)
06

Ongoing Support & Partnership

Optional · monthly retainer

After handoff, clients may elect to retain us for continued support, feature work, or scheduled architecture reviews. Retainers are sized to the actual operating reality of the system. We will not sell you a retainer larger than you need.

  • Monthly retainer for bug fixes, security patching, and dependency updates
  • Quarterly architecture and cost reviews
  • On-call support tier for production incidents (where engaged)
  • Feature work billed against an agreed budget
  • Clean exit at any month, with full documentation maintained
Engagement Mechanics

The operational parts of working with us.

These are the things that aren't on a project timeline but make the difference between a clean engagement and a painful one. We document them in advance.

Client Onboarding & Verification

New business clients complete a brief onboarding process before kickoff. This includes a Know-Your-Business (KYB) verification step in line with the firm's compliance posture: confirmation of the client's legal entity, business address, and an authorized signatory.

The verification is light-touch — typically a single document upload and a short questionnaire — and is completed in parallel with the discovery phase so it does not delay project start. We use the same standard for every new client.

Scope Changes & Change Orders

Scope will change. We treat that as normal. The mechanism is a written change order: a one-page document that names the requested change, the effort impact, the schedule impact, and the cost impact. Nothing material moves without one.

Small course corrections inside the backlog do not require paperwork. The threshold for a formal change order is documented in the engagement letter so there is no ambiguity about when one applies.

Code Ownership & IP Transfer

All work product — source code, infrastructure definitions, documentation, and design artifacts — transfers to the client on payment. There are no licensing strings, no proprietary "platform" you have to keep paying for, no source-code escrow theatre.

The firm retains the right to describe the engagement in general terms (industry, scope, technologies used) for portfolio purposes, only with explicit client consent. Anything more specific requires written approval.

Security & Confidentiality

A mutual NDA is part of every engagement. Internally we apply least-privilege access to client environments, use hardware-backed credentials, and rotate secrets on a schedule. Client data is never used to train or evaluate third-party services without explicit consent.

For regulated work — healthcare, financial services, government — we are willing to sign sector-specific agreements (HIPAA BAAs, etc.) where the work justifies it, and to staff the engagement accordingly.

Ready to Begin

A short call, then a written brief.

That's how every engagement begins. The discovery phase is fixed-fee and small enough that we can find out together whether we're the right firm for the work.
